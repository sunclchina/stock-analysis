## 二、推荐的开发顺序（前后端并行策略）

### 2.1 阶段划分

```
阶段1 ─── 基础骨架（3~5天）
├── 前端：M00 统一导航菜单 + 路由框架 + 主题系统
├── 后端：API网关骨架 + 数据源适配层（通达信本地解析）
└── 数据：数据源基础解析脚本

阶段2 ─── 核心数据模块（5~7天）
├── 前端：M06 系统配置页面（7标签页）
├── 后端：M06 配置API + M02 行情数据接口
└── 数据：实时行情数据采集管道

阶段3 ─── 预警+仪表盘（4~6天）
├── 前端：M03 预警列表页 + M01 仪表盘
├── 后端：M03 预警计算引擎 + M01 仪表盘聚合API
└── 数据：预警指标计算

阶段4 ─── 选股+分析（5~7天）
├── 前端：M04 选股页面 + M05 分析页面
├── 后端：M04 选股引擎 + M05 分析引擎（大模型调用）
└── 集成：AI大模型接口对接

阶段5 ─── 联调+优化（3~5天）
├── 全模块联调测试
├── 性能优化（缓存/频率控制）
└── 容错处理（数据源降级/异常断线恢复）
```

### 2.2 并行策略说明

| 并行组 | 前端 | 后端 | 数据层 |
|--------|------|------|--------|
| 组A（阶段1） | 导航菜单UI | API网关 + 数据源 | 通达信解析 |
| 组B（阶段2） | 配置页UI | 配置API + 行情API | 行情采集 |
| 组C（阶段3） | 预警+仪表盘UI | 预警引擎 + 聚合API | 预警数据 |
| 组D（阶段4） | 选股+分析UI | 选股引擎 + AI分析 | 大模型接口 |
| 组E（阶段5） | 联调优化 | 联调优化 | 容错降级 |

**关键依赖关系：**
- M01 仪表盘 → 依赖 M02/M03/M05/M06 接口就绪（阶段3之后）
- M04 选股 → 依赖 M03 预警指标（阶段3之后）
- M05 分析 → 依赖 M03/M04 数据 + AI接口（阶段4）

## 三、技术栈选型建议

### 3.1 总体架构

```
┌────────────────────────────────────────────────────────┐
│                    前端层 (Port 8080)                    │

└──────────────────┬─────────────────────────────────────┘
                   │ HTTP REST + WebSocket
┌──────────────────▼─────────────────────────────────────┐
│                   API网关层                              │
│  Python FastAPI / Node.js Express                       │
│  WebSocket Server + Rate Limiter + Auth                 │
└──────┬──────────────────────┬─────────────────────────┬─┘
       │                      │                         │
┌──────▼──────┐    ┌─────────▼────────┐    ┌───────────▼──┐
│ 业务逻辑层   │    │ 数据计算/分析层   │    │  AI服务层    │
│ FastAPI      │    │ Pandas/Numpy     │    │ DeepSeek API │
│ 模块路由     │    │ 预警引擎/选股引擎  │    │ 提示词模板   │
└──────┬──────┘    └──────────────────┘    └──────────────┘
       │
┌──────▼──────────────────────────────────────────────┐
│                 数据服务层                             │
├──────────────────┬──────────────────┬────────────────┤
│  本地通达信      │  网络公开数据源    │  本地缓存      │
│  ds_stk.dat解析  │  新浪/东方财富    │  SQLite/Redis  │
│  .day文件解析    │  Baostock+       │  日线/分钟线    │
│  TFZ/DAT解析     │  网页爬取(应急)   │  配置+模板     │
└──────────────────┴──────────────────┴────────────────┘
```

### 3.2 技术栈详表

#### 前端

| 技术 | 选型 | 理由 |
|------|------|------|
| UI框架 | React 18 + TypeScript | 生态丰富、组件库成熟、TS提升可维护性 |
| 构建工具 | Vite 6 | 极速HMR、开箱TS支持、易于配置8080端口 |
| UI组件库 | Ant Design 5 | 金融系统常用，表格/表单/布局完善 |
| 图表库 | ECharts 5 | K线/分时图/仪表盘原生支持 |
| 状态管理 | Zustand | 轻量、无样板代码、支持中间件 |
| 路由 | React Router 7 | 嵌套路由/懒加载完善 |
| WebSocket | socket.io-client | 实时行情/预警推送 |
| HTTP | axios | 拦截器/取消请求/重试 |
| 主题 | CSS Variables + Ant Design ConfigProvider | 深色/浅色模式切换 |
| 样式方案 | Tailwind CSS 4 | 原子化CSS、响应式断点 |

#### 后端

| 技术 | 选型 | 理由 |
|------|------|------|
| Web框架 | Python FastAPI | 异步支持、自动OpenAPI文档、类型校验 |
| ASGI服务器 | Uvicorn | 高性能异步服务器 |
| ORM | SQLAlchemy 2.0 + async | 异步数据库操作 |
| 数据库 | SQLite（轻量）/ PostgreSQL（生产） | 个人系统SQLite够用 |
| 缓存 | Redis / 内存缓存 | 近期数据缓存、预警状态 |
| WebSocket | FastAPI WebSocket | 原生支持、与REST共用路由 |
| 任务调度 | APScheduler | 定时计算（盘前/盘中/盘后） |
| 数据计算 | Pandas + NumPy | 技术指标计算、数据分析 |
| HTTP客户端 | httpx | 异步HTTP请求（数据源/大模型） |
| 配置管理 | Pydantic Settings | .env + 系统配置模块联动 |

#### 数据层

| 组件 | 选型 | 说明 |
|------|------|------|
| 主力数据源 | 通达信本地(DS_STK.DAT + .day) | 二进制解析 |
| 备用数据源 | pytdx / Sina / EastMoney | 网络API |
| 历史数据 | Baostock+ | 日线/分钟线批量 |
| 本地缓存 | SQLite | 近期数据缓存 |
| 配置存储 | .env + SQLite | 环境变量 + 数据库配置 |
| 模板存储 | JSON/MD文件 | 分析模板文件系统 |

### 3.3 API风格

- **数据接口**：RESTful JSON API，统一前缀 `/api/v1`
- **实时推送**：WebSocket，协议 `ws://localhost:8080/ws`
- **文件下载**：静态文件服务 `GET /api/v1/download/{filename}`
- **认证**：基础API密钥（个人系统，简化为请求头传递）

## 五、数据流图文字说明

### 5.1 盘前数据流

```
[通达信本地数据] → 加载昨日行情/日线 → 缓存至SQLite
       ↓
[新浪/东方财富] → 盘前新闻/政策 → AI大模型
       ↓
[AI大模型] → 按盘前提示模板 → 盘前提示卡片
       ↓
[仪表盘M01] → 盘前提示展示
[实时行情M02] → 归档盘前提示
```

### 5.2 盘中实时数据流

```
定时循环（默认5s/次）：
  [数据源(通达信/网络)] → 实时行情快照
       ↓
  [预警引擎M03] → 7大模块并行计算（多线程，每只股票独立）
       ├── 价格预警 ← 实时价 + 日线基准
       ├── 涨跌预警 ← 开盘价 + 实时价
       ├── 趋势预警 ← 均线系统 + 突破判断
       ├── 共振预警 ← 8项指标计分
       ├── 财务预警 ← 每日加载一次（非实时）
       ├── 突发预警 ← 加权计分
       └── 风险评分 ← 基础分 + 动态分
       ↓
  [综合决策矩阵] → 综合颜色 + 操作建议
       ↓
  [事件推送(WS)] → 仅颜色变化时推送到前端
       ↓
  [前端M02列表] → 行情列表更新 + 预警高亮
  [前端M03预警] → 预警汇总更新 + 角标
  [前端M01仪表盘] → 监控卡片更新 + 预警汇总更新
```

### 5.3 选股数据流

```
全A股（~5000只）
  ├── 第1层：快速底层过滤 → 剔除ST/停牌/流动性差 → ~3000只
  ├── 第2层：技术面粗筛 → 6选≥4 → ~300只
  ├── 第3层：深度技术精筛 → 强制条件+附加项 → ~80只
  ├── 第4层：财务+事件过滤 → 财务安全+无风险 → ~30只
  └── 第5层：综合评分排序 → ≥85分+容量控制 → ≤20只
       ↓
  [使用预警模块M03提供的指标字段]
       ↓
  输出 → 前端选股列表 → 可加入自选/监控池/导入分析
```

### 5.4 分析报告数据流

```
[用户选择分析类型]
   ├── 大盘复盘 → 输入日期+关注股票(≤10) → 调用行情+预警数据
   ├── 个股分析 → 输入股票代码 → 调用行情+财务+技术指标
   └── 批量分析 → 输入股票列表(≤10) → 逐只分析+汇总
         ↓
   [系统收集数据] → 整合行情/财务/技术/事件数据
         ↓
   [AI大模型] → 按模板生成结构化报告 → 返回完整文本
         ↓
   [前端展示] → 富文本渲染（折叠/展开/跳转/图标标记）
         ↓
   [下载] → TXT/Markdown格式 → 按规范命名一键下载
```

### 5.5 配置管理数据流

```
[用户操作配置页M06]
   ├── 系统设置 → 读写.env → 即时生效/提示重启
   ├── 自选股管理 → 存储到SQLite → 同步到监控池/选股/行情
   ├── 监控池管理 → 存储到SQLite → 同步到行情/预警
   ├── 数据源管理 → 状态检测 → 自动降级策略
   ├── 模板管理 → 文件存储 → 同步到选股/行情/分析
   ├── 用户偏好 → 存储到SQLite/Session → 即时生效
   └── 系统状态 → 读取系统资源 → 实时展示
         ↓
   [配置变更事件] → 通过WebSocket推送到前端各模块
```

## 七、关键设计决策

### 7.1 技术选型依据

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 前端框架 | React vs Vue | React | 更成熟的TypeScript生态，Ant Design金融级组件 |
| 后端框架 | Python vs Node.js | Python FastAPI | 数据计算库(Pandas)生态优于Node；AI集成更自然 |
| 数据库 | SQLite vs PostgreSQL | SQLite（初期） | 个人系统单机使用，0运维成本；通过SQLAlchemy可平滑迁移 |
| 实时通信 | WS vs SSE vs Polling | WebSocket | 双向通信，适合预警推送+行情更新 |
| 数据缓存 | Redis vs Memory | Memory Cache（初期） | 单机无需Redis，Python dict+cachetools足够 |

### 7.2 实时性保障策略

1. **高频数据本地计算**：预警引擎直接读取通达信本地缓存，避免网络延迟
2. **变化驱动推送**：仅预警颜色变化时推送，避免无谓带宽消耗
3. **动态频率调整**：正常5秒→预警1秒→停牌暂停
4. **多线程并行**：10线程×每线程10只股票，单次全量计算≤2秒

### 7.3 容错与降级策略

1. **数据源降级**：连续3次失败自动切换备用源，页面只做状态监控
2. **AI模型降级**：大模型调用失败→自动重试3次→返回"分析失败，请重试"
3. **模块故障隔离**：一个模块异常不影响其他模块
4. **配置校验**：实时校验端口/URL/数值范围，高危操作二次确认

### 7.4 个性化与多端适配

1. **主题体系**：CSS Variables实现深色/浅色/自动模式，全局即时切换
2. **响应式断点**：≥1200px桌面 / 768~1199px平板 / <768px移动端
3. **布局自定义**：仪表盘卡片可拖拽排序/隐藏

## 八、后续实施建议

1. **先打骨架再填肉**：阶段1确保前后端通信链路畅通，再逐步添加业务逻辑
2. **数据源先行**：通达信本地解析是整个系统的基础，需优先完成并测试
3. **预警引擎最核心**：选股和分析两个模块都依赖预警指标字段，预警引擎应优先实现
4. **AI集成最后**：智能分析模块依赖其他模块的数据接口，建议在数据流稳定后再接入
5. **逐步替换模拟数据**：阶段1允许用测试数据打通流程，但阶段2后必须全部替换为实时数据
